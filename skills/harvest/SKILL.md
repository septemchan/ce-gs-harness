# /ce-gs-harness:harvest — Generate or update `.claude/CLAUDE.md`

Scan design docs and project files, then produce a CLAUDE.md that tells Claude what it can't figure out on its own.

## Core principle

CLAUDE.md is not a project profile. It's a memo of things Claude cannot infer by reading the code.

Every line should pass the **one-line test**: if you delete this line, will Claude make a mistake? If not, don't include it. The reason this matters: CLAUDE.md loads into context at the start of every session. Bloated files waste tokens and cause Claude to silently ignore rules buried deep in the text. 60 lines of high-signal content outperforms 200 lines of documentation.

**Belongs in CLAUDE.md:**
- Commands that aren't obvious from config files (which package manager, which test runner flags, custom scripts)
- Design decisions invisible in code (why this architecture, data flow direction, module ownership boundaries)
- Non-obvious conventions (naming patterns that differ from framework defaults, team-agreed file organization rules)
- Constraints and gotchas (minimum runtime versions, env var requirements, known footguns)
- Operational procedures: step-by-step chains, decision trees, conditional trigger rules (see below)

**Does NOT belong:**
- Tech stack basics readable from config files (language, framework, dependency list)
- Directory structure Claude can explore with `ls`
- Code style enforced by linters or formatters
- Standard language/framework conventions Claude already knows

### Facts vs procedures

The one-line test applies differently depending on the type of content:

| Type | Examples | Filtering strategy |
|------|----------|-------------------|
| **Facts** | Tech stack, directory structure, dependency list, config values | Filter aggressively — Claude can read files and infer these |
| **Procedures** | Step-by-step chains, decision trees, conditional trigger rules, checklists with specific conditions | Filter conservatively — keep even if detailed, because each step is a point where Claude could make a mistake |

A 10-step recovery chain looks verbose, but if each step maps to a different action ("find X but not Y → do Z"), compressing it into "scan backwards" loses the decision logic Claude needs. Procedures are high-value precisely because they encode judgment calls that aren't obvious from the code.

When in doubt: ask whether the content is a **fact Claude can look up** or a **procedure Claude must follow**. Facts compress well; procedures usually don't.

## Two modes

### Mode A — Design docs exist

Triggered when any of `docs/brainstorms/`, `docs/plans/`, or `docs/ideation/` contains files.

1. Read the most recent files in those directories (sort by modification time, take the latest 3-5 files)
2. Scan project config files to identify available commands and non-obvious stack choices:
   - `package.json`, `tsconfig.json`, `pyproject.toml`, `go.mod`, `requirements.txt`, `Cargo.toml`, `Gemfile`, `Makefile`, `docker-compose.yml`, `build.gradle`
3. Check for design spec files: `DESIGN.md` in project root, or `design-system/*/MASTER.md` (/ui-ux-pro-max output). Note the actual file path if found — it will be injected into the 约定 section after filtering (see Output structure)
4. Determine the primary source directory: if `.claude/CLAUDE.md` already exists and declares a source code parent directory (e.g., `apps/`), use that; otherwise fall back to `src/` or the most obvious source directory. Scan it (limit 2 levels deep — deeper scans on large codebases produce too much noise for a summary document)
5. Filter all collected information through the one-line test, then generate CLAUDE.md

### Mode B — No design docs (fallback)

Triggered when none of the design doc directories exist or they are empty.

1. Scan project config files only (same list as Mode A)
2. Check for design spec files (same search as Mode A step 3)
3. Scan root directory and primary source directory structure (if `.claude/CLAUDE.md` declares a source code parent directory, use that; otherwise `src/`) (limit 2 levels deep)
4. Generate CLAUDE.md from inferred data, being extra conservative — less context means more risk of wrong guesses, so prefer placeholders over speculation
5. Add at top of file: `<!-- Generated by /harvest (mode B). Review and refine. -->`

## Output structure

4 sections with Chinese headings. Target 50-80 lines; hard cap at 100 lines (beyond ~100 lines Claude starts ignoring rules at the bottom of the file).

<example title="generated-CLAUDE.md">
```markdown
# 技术栈

(Only what Claude can't infer from config files. Skip the obvious — e.g., "uses React" when package.json says so. Focus on: package manager preference, runtime version constraints, non-obvious build choices)

# 命令

(Highest-value section. Claude frequently guesses wrong about which commands to run. Be specific: `pnpm dev` not "run the dev server". Only include commands confirmed in config files or scripts)

# 架构

(Skip directory listings — Claude can `ls`. Focus on what's invisible in code: design decisions and rationale, data flow between modules, module responsibilities, key patterns and why they exist)

# 约定

(Non-obvious conventions only. Skip anything enforced by linters/formatters or standard to the framework. Naming deviations, team-agreed organization rules, commit/PR conventions)
```
</example>

### Design spec reference injection

If a design spec file was found during scanning (step 3 in Mode A, step 2 in Mode B), inject a constraint line at the end of the 约定 section. Like the footer comment, this line is injected **after** one-line test filtering — it is a procedural constraint, not an inferable fact.

Template: `All UI must follow {actual file path}. Do not invent new colors/typography outside the spec.`

Examples:
- If `DESIGN.md` found in root: `All UI must follow DESIGN.md. Do not invent new colors/typography outside the spec.`
- If `design-system/my-app/MASTER.md` found: `All UI must follow design-system/my-app/MASTER.md. Do not invent new colors/typography outside the spec.`

If both `DESIGN.md` and `design-system/*/MASTER.md` exist, use `DESIGN.md` (root file takes precedence). If multiple `MASTER.md` files match the glob (monorepo with multiple apps), reference the first match and note the others exist.

End the file with a footer comment:

```
<!-- Generated by /harvest (mode A|B), YYYY-MM-DD -->
```

If a section has no high-value content after filtering, keep the heading and insert:

```
<!-- No non-obvious info to add — Claude can infer from code -->
```

This is better than filling a section with low-value content just to avoid emptiness.

## Update behavior

When `.claude/CLAUDE.md` already exists:

1. Read the existing file completely
2. Generate a new version from current sources
3. Present changes in a before/after comparison, section by section, so the user can see exactly what changed in each section
4. Ask the user: **accept all / select by section / cancel**
5. Preserve any sections with headings NOT in the 4-section template (user-added custom sections) — append them after the standard sections
6. Update the bottom footer comment with the current date

## Workflow

1. **Detect mode** — check for `docs/brainstorms/`, `docs/plans/`, `docs/ideation/`
2. **Collect information** — read config files, scan directories, read design docs (Mode A only)
3. **Filter** — for each piece of information, apply the one-line test: can Claude figure this out by reading the code? If yes, drop it
4. **Check for existing `.claude/CLAUDE.md`**
5. **Generate content** — build the 4 sections from filtered information
6. **Present to user**:
   - First time: show generated content, ask for confirmation, then write
   - Update: show before/after by section, ask for confirmation (accept all / select by section / cancel)
7. **Write file** — ensure `.claude/` directory exists, write CLAUDE.md

## Edge cases

- **Monorepo**: Root `package.json` with a `workspaces` field is a workspace root, not the app itself. Scan workspace packages to find actual tech stacks, list per-package in 技术栈 section
- **Multiple tech stacks**: When a project has both `package.json` and `pyproject.toml` (or similar), identify which is primary and note how they relate
- **Empty project**: No config files and no source directory — generate Mode B with all sections as placeholders. Don't invent content
- **Secrets**: Never read `.env` file values, only key names from `.env.example`. Reason: `.env` may contain real credentials. If `.env.example` doesn't exist, don't reference env vars at all
- **Unverified info**: Only include commands, URLs, and file paths that are confirmed to exist in the project. If a URL or install command isn't found in README, config files, or scripts, don't invent one
- **Design spec file**: If `DESIGN.md` or `design-system/*/MASTER.md` exists, inject the constraint line in 约定 section (see "Design spec reference injection" above). Do not inject if the file does not exist — never reference a design spec that hasn't been created yet

## Gotchas

Known failure patterns from testing. Check for these in generated output:

- **URL/命令编造**: Claude 在 Mode B（信息少）时倾向编造 git clone URL 或安装命令来填充命令 section。如果项目 README 或配置文件里没有这个命令，不要生成它
- **操作规程过度压缩**: Claude 倾向把 10 步决策链压成一句概括。对照 "Facts vs procedures" 段检查：如果每步对应不同操作，必须逐步保留
- **低价值填充**: 当某个 section 信息不足时，Claude 倾向用泛泛的描述填满它（如"项目结构清晰，模块划分合理"），而不是使用占位注释。空 section 用 `<!-- No non-obvious info -->` 比低价值内容好

## Important notes

- Use Chinese for section headings (技术栈, 命令, 架构, 约定)
- CLAUDE.md describes the project as-is, not aspirational goals from design docs
- When in doubt between including and excluding a line, exclude it — the user can always add it back, but bloat is hard to notice and fix
