# /harvest — Generate or update `.claude/CLAUDE.md`

Scan design docs and project files, then produce a concise CLAUDE.md that describes the project as-is.

## Two modes

### Mode A — Design docs exist

Triggered when any of `docs/brainstorms/`, `docs/plans/`, or `docs/ideation/` contains files.

1. Read the most recent files in those directories (sort by modification time, take the latest 3-5 files)
2. Scan project config files to determine tech stack:
   - `package.json`, `tsconfig.json`, `pyproject.toml`, `go.mod`, `requirements.txt`, `Cargo.toml`, `Gemfile`
3. Scan `src/` directory structure (limit 2 levels deep)
4. Generate CLAUDE.md combining doc insights + project data

### Mode B — No design docs (fallback)

Triggered when none of the design doc directories exist or they are empty.

1. Scan project config files only (same list as Mode A)
2. Scan root directory and `src/` directory structure (limit 2 levels deep)
3. Generate CLAUDE.md from inferred data
4. Add at top of file: `<!-- Generated by /harvest (mode B). Review and refine. -->`

## Output structure

CLAUDE.md has exactly 4 sections with Chinese headings. Keep total length under 100 lines.

<example title="generated-CLAUDE.md">
```markdown
# 技术栈

(Language, framework, major dependencies, build tools)

# 命令

(Dev server, build, test, lint, deploy — only commands confirmed in config files or scripts)

# 架构

(Directory layout, key modules, data flow — sourced from src/ scan and design docs)

# 约定

(Coding conventions, naming patterns, file organization rules — sourced from design docs and existing code patterns)
```
</example>

End the file with a footer comment:

```
<!-- Generated by /harvest (mode A|B), YYYY-MM-DD -->
```

If a section cannot be inferred with confidence, insert a placeholder:

```
<!-- Could not infer — fill in manually -->
```

## Update behavior

When `.claude/CLAUDE.md` already exists:

1. Read the existing file completely
2. Generate a new version from current sources
3. Show the diff to the user (old vs new, section by section)
4. Ask the user: **accept all / select by section / cancel**
5. Preserve any sections with headings NOT in the template (user-added custom sections) — append them after the 4 standard sections
6. Update the bottom footer comment with the current date

## Workflow

1. **Detect mode** — check for `docs/brainstorms/`, `docs/plans/`, `docs/ideation/`
2. **Collect information** — read config files, scan directories, read design docs (Mode A only)
3. **Check for existing `.claude/CLAUDE.md`**
4. **Generate content** — build the 4 sections
5. **Present to user**:
   - First time: show generated content, ask for confirmation, then write
   - Update: show diff, ask for confirmation (accept all / select by section / cancel)
6. **Write file** — ensure `.claude/` directory exists, write CLAUDE.md
7. **Verify** — confirm file is under 100 lines

## Important notes

- Never read `.env` file values — only read key names from `.env.example`
- Use Chinese for section headings (技术栈, 命令, 架构, 约定)
- CLAUDE.md describes the project as-is, not aspirational goals
- If information cannot be determined, use a placeholder comment rather than guessing
- Monorepo root `package.json` may contain workspace tools, not the actual tech stack — check for `workspaces` field and scan workspace packages instead
- Limit `src/` scan to 2 levels deep to avoid overexpanding large codebases
