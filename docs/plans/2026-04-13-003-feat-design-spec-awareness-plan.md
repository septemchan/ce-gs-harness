# feat: Add design spec file awareness to harness-audit and harvest

## Overview

让 harness-audit 和 harvest 感知前端项目中的设计规范文件（DESIGN.md 或 MASTER.md），在项目健康度评估中检测其存在性，在 CLAUDE.md 生成时自动引用。

## Problem Frame

前端项目可能存在两种设计规范文件：DESIGN.md（Google Stitch 标准，项目根目录）和 MASTER.md（/ui-ux-pro-max 输出，`design-system/[项目名]/` 下）。目前 harness-audit 不检测这些文件，harvest 生成 CLAUDE.md 时也不引用它们，导致 AI 可能忽略已有的设计规范。(see origin: `docs/brainstorms/2026-04-13-design-md-and-agents-md-awareness-requirements.md`)

## Requirements Trace

- R1. harness-audit Layer 1 新增设计规范文件检查（Optional ×0.5，仅前端项目）
- R2. 缺失时建议 /ui-ux-pro-max 生成
- R3. harvest 检测到设计规范文件时，在 CLAUDE.md 约定 section 注入硬约束引用行

## Scope Boundaries

- 不生成设计规范文件内容
- 不验证设计规范文件质量
- 不改评分公式逻辑
- 不改 workflow-map 或其他 rules
- 不涉及 AGENTS.md

## Context & Research

### Relevant Code and Patterns

**harness-audit 现有 Optional 检查项模式** (`skills/harness-audit/SKILL.md:59-63`)：
```
#### Optional (×0.5)

**Workflow entry points** _(skills/ or commands/)_
- Pass: project has at least 1 skill or 1 command
```
新检查项沿用此格式，加条件触发逻辑（参考 Product-Spec.md 的条件跳过模式，`SKILL.md:49-52`）。

**harness-audit 修复建议模式** (`skills/harness-audit/SKILL.md:158-163`)：
```
- Missing/stale CLAUDE.md → `/ce-gs-harness:harvest`
- Missing Product-Spec.md → `/ce-gs-harness:product-spec-draft`
```
新建议沿用此格式。

**harvest 的 footer 注入模式** (`skills/harvest/SKILL.md:96-97`)：
```
End the file with a footer comment:
<!-- Generated by /harvest (mode A|B), YYYY-MM-DD -->
```
设计规范引用行使用相同的"过滤后注入"机制。

**harvest 的 约定 section** (`skills/harvest/SKILL.md:89-91`)：
```
# 约定
(Non-obvious conventions only.)
```
引用行插入此 section。

## Key Technical Decisions

- **前端检测策略**：复用 package.json dependencies/devDependencies 检查 + 前端配置文件存在性检查。不定义穷举列表，给出代表性框架名并标注"等常见前端框架"，实现者可自行扩展 (see origin)
- **搜索路径优先级**：先查 `DESIGN.md`（项目根目录），再查 `design-system/*/MASTER.md`（glob 匹配项目名子目录），找到任一即为 Pass (see origin)
- **引用行注入时机**：引用行注入 约定 section 末尾。与 footer 注释一样，引用行不受 one-line test 过滤（因为它是程序性约束而非可推断的事实）。理由：Claude 可以 `ls` 发现文件存在，但不会自动将其视为强制约束 (see origin)
- **引用行包含实际路径**：如果找到的是 `design-system/my-project/MASTER.md`，引用行应写 `All UI must follow design-system/my-project/MASTER.md.`，不用泛化路径

## Open Questions

### Deferred to Implementation

- harvest update mode 重复运行时引用行是否会重复注入：现有的 update 流程（重新生成 + before/after 对比）应该自然处理，但实现时需确认幂等性

## Implementation Units

- [x] **Unit 1: harness-audit — 新增设计规范文件检查**

**Goal:** 在 Layer 1 的 Optional 检查项中新增设计规范文件存在性检查

**Requirements:** R1, R2

**Dependencies:** None

**Files:**
- Modify: `skills/harness-audit/SKILL.md`

**Approach:**
- 在 `#### Optional (×0.5)` section 中，现有 "Workflow entry points" 检查项之后，新增 "Design spec file" 检查项
- 条件触发逻辑：描述前端项目检测条件（package.json 含前端框架依赖，或存在前端配置文件），非前端项目跳过
- 搜索路径：`DESIGN.md`（根目录）→ `design-system/*/MASTER.md`
- Pass/Fail 条件：找到任一为 Pass
- 在 `## Important notes` section 的修复建议列表中新增一条：`Missing design spec → /ui-ux-pro-max`

**Patterns to follow:**
- Product-Spec.md 的条件检测格式 (`SKILL.md:49-52`)：`_(conditional — skip for ...)_` + Detection 逻辑说明
- Workflow entry points 的 Optional 检查项格式 (`SKILL.md:59-63`)
- 修复建议列表格式 (`SKILL.md:158-163`)

**Test scenarios:**
- Happy path: 前端项目根目录有 DESIGN.md → Pass，扣分正确（×0.5 权重）
- Happy path: 前端项目有 design-system/my-app/MASTER.md → Pass
- Happy path: 非前端项目（无 package.json 或无前端依赖）→ 检查被跳过，不出现在结果中
- Edge case: 同时存在 DESIGN.md 和 MASTER.md → Pass（找到任一即可）
- Edge case: 前端项目但无设计规范文件 → Fail，显示修复建议 /ui-ux-pro-max

**Verification:**
- SKILL.md 中 Optional section 新增了设计规范检查项，包含条件触发和搜索路径说明
- Important notes 的修复建议列表包含 design spec 条目

- [x] **Unit 2: harvest — 设计规范文件引用注入**

**Goal:** harvest 生成 CLAUDE.md 时检测设计规范文件并在约定 section 注入硬约束引用行

**Requirements:** R3

**Dependencies:** None（与 Unit 1 独立，可并行）

**Files:**
- Modify: `skills/harvest/SKILL.md`

**Approach:**
- 在 Mode A 和 Mode B 的扫描步骤（step 2）中，增加一步：检测设计规范文件是否存在（搜索路径同 Unit 1）
- 在 "Output structure" section 的 约定 部分说明：如果检测到设计规范文件，在约定 section 末尾注入引用行。与 footer 注释一样，引用行不受 one-line test 过滤（程序性约束，非可推断事实）
- 引用行模板：`All UI must follow {实际文件路径}. Do not invent new colors/typography outside the spec.`
- 在 "Edge cases" section 新增一条：设计规范文件存在时的引用注入行为

**Patterns to follow:**
- Footer 注释的后置注入模式 (`SKILL.md:96-97`)
- Edge cases section 的格式 (`SKILL.md:132-136`)

**Test scenarios:**
- Happy path: 项目根目录有 DESIGN.md → 约定 section 包含 `All UI must follow DESIGN.md.` 引用行
- Happy path: 项目有 design-system/my-app/MASTER.md → 约定 section 包含带完整路径的引用行
- Happy path: 无设计规范文件 → 约定 section 不受影响
- Edge case: harvest update mode 重复运行 → 引用行不重复（重新生成自然处理）

**Verification:**
- SKILL.md 的 Mode A/B 扫描步骤中提及设计规范文件检测
- Output structure 的约定 section 说明了引用行注入逻辑
- Edge cases 中包含设计规范文件相关条目

## System-Wide Impact

- **Interaction graph:** harness-audit 的评分分母会因新增 Optional 检查项而微增（+0.5），现有项目百分比分数略微下降，影响极小
- **API surface parity:** harness-audit Init mode 目前不受影响（Init mode 的推荐流程不变），但未来可考虑在 Init mode 也提及设计规范文件
- **Unchanged invariants:** Layer 1 其他检查项、Layer 2 质量检查、评分公式、maturity level 阈值均不变

## Risks & Dependencies

| Risk | Mitigation |
|------|------------|
| 前端检测可能漏掉非主流框架 | 检查条件用"等常见前端框架"留扩展口，Optional 权重降低误判影响 |
| harvest 引用行在 update mode 可能重复 | harvest 的 update 流程是完整重新生成 + 对比，引用行随每次生成自然幂等 |
