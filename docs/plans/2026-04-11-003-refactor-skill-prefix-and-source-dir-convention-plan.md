# refactor: Skill 全称前缀 + 源代码目录约定

## Overview

所有 SKILL.md 和 workflow-map.md 中的 harness skill 引用从短名称（`/harvest`）改为全称（`/ce-gs-harness:harvest`），解决 AI 调用时不带前缀的问题。同时新增源代码目录约定 rule。

## Problem Frame

AI 读到 SKILL.md trigger 文本和 workflow-map.md 中的短名称后，倾向用短名称调用 skill，导致前缀丢失。源代码目录没有约定，多端项目子项目散在根目录。(see origin: docs/brainstorms/2026-04-11-skill-prefix-and-source-dir-convention-requirements.md)

## Requirements Trace

- R1. SKILL.md frontmatter trigger 文本统一使用 `/ce-gs-harness:skill-name` 全称
- R2. SKILL.md 正文中的 harness skill 引用统一使用全称
- R3. workflow-map.md 中的 harness skill 引用统一使用全称
- R4. 生成内容中的归属注释保留短名称
- R5. 新增 rule：多端/多模块项目子项目必须收在同一个根目录下属文件夹
- R6. 父目录名称不固定，brainstorm/plan 阶段确定
- R7. harvest skill 扫描源代码时适配此约定

## Scope Boundaries

- 不改 CE / gstack 插件的引用
- 不改 plugin.json
- SKILL.md 中的描述性引用（如 `product-spec-draft's job`）不带斜杠，保持原样
- harvest 生成模板中的 `<!-- Generated by /harvest -->` 保留短名称 (R4)

## Key Technical Decisions

- 只改带 `/` 的 slash command 格式引用，不改描述性文本中的 skill 名称：因为 AI 只会把 `/xxx` 格式当作调用指令
- `/update-config` 是 gstack skill，不改

**Approach:**

**新建 rule:**
规则内容：多端/多模块项目的所有子项目（前端、后端、小程序端等）必须收在同一个根目录下属文件夹内。名称在 brainstorm/plan 阶段根据项目性质确定（常见选择：`apps/`、`packages/`、`services/`），确定后写进项目 CLAUDE.md 的架构 section。

Why：防止子项目散在根目录导致项目结构混乱。

When to skip：单端项目（只有一个 `src/`），或框架已约定目录结构的项目。

**harvest 适配 (R7):**
在 harvest SKILL.md 的 Mode A 和 Mode B 扫描步骤中，增加一步：如果 `.claude/CLAUDE.md` 已声明源代码父目录，优先扫描该目录而非默认的 `src/`。这是最小改动，不改变 harvest 的核心逻辑。

具体位置：
- Mode A step 3（line 57）：`Scan src/ or primary source directory` → 增加"先读已有 CLAUDE.md 确认源代码父目录"
- Mode B step 2（line 65）：同理

**Test expectation:** none — rule 是纯文档，harvest 改动是扫描路径的文字描述调整
