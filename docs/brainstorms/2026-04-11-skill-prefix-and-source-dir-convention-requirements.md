# Skill 全称前缀 + 源代码目录约定

## Problem Frame

两个独立问题，都影响 ce-gs-harness 在实际使用中的一致性：

1. **Skill 名称不带前缀**：所有 SKILL.md 的 trigger 文本和 workflow-map.md 的引用都写的是短名称（如 `/harvest`），导致 AI 调用时不带 `ce-gs-harness:` 前缀。虽然系统注册的全称是 `ce-gs-harness:harvest`，但 AI 读到短名称就倾向用短名称。
2. **源代码目录散乱**：多端项目（前端/后端/小程序端）的子项目各自散在根目录，没有统一的父目录约定。

## Requirements

**Skill 前缀一致性**

- R1. 所有 SKILL.md frontmatter 的 trigger 文本中，`/skill-name` 格式统一改为 `/ce-gs-harness:skill-name`
- R2. 所有 SKILL.md 正文中引用自己或其他 harness skill 的地方，统一使用 `/ce-gs-harness:skill-name` 全称
- R3. workflow-map.md 中所有 harness skill 引用统一使用全称
- R4. 生成内容中的归属注释（如 `<!-- Generated by /harvest -->`）保留短名称，因为这是文档标记不是调用指令

**源代码目录约定**

- R5. 新增一条 rule，约定：多端/多模块项目的子项目必须收在同一个根目录下属文件夹里，不允许各自散在根目录
- R6. 父目录名称不固定，在 brainstorm/plan 阶段根据项目性质确定，确定后写进项目的 CLAUDE.md
- R7. harvest skill 扫描源代码目录时，适配此约定（扫描项目级 CLAUDE.md 中声明的源代码父目录）

## Success Criteria

- AI 调用 harness skill 时使用全称（如 `/ce-gs-harness:harvest`）
- 新项目规划时 AI 主动确认源代码父目录名称，而不是让子项目散在根目录

## Scope Boundaries

- 不改 CE 插件或 gstack 插件的 skill 引用（它们有自己的前缀体系）
- 不改 plugin.json 或 skill 注册机制（前缀生成本身没问题）
- R7 是最小改动：harvest 读已有 CLAUDE.md 中的目录声明，不自己发明扫描逻辑

## Key Decisions

- 前缀全称 vs 短名称：选全称，因为短名称导致 AI 混用
- 父目录名称固定 vs 不固定：选不固定，因为 `app/` 等名称在某些框架中有特殊含义，强绑会冲突
